# 1. UI Layer & App Shell

**Scope:** `src/components/`, `src/hooks/`, `src/App.tsx`, `src/main.tsx`

## Overview

This subsystem is the entire UI layer of the Viasox Marketing Intelligence SPA — a single-page React 18 + TypeScript + Tailwind v4 app with no router. `src/main.tsx` mounts `<App/>` (plus two side jobs: installing `window.__brain` console dev-tools and pre-warming the IndexedDB-backed VoC index). `src/App.tsx` is the app's only state machine: a `view` state of type `AppView` ('auth' → 'apikey' → 'upload' → 'processing' → 'dashboard' → 'module') plus an `activeModule: ModuleId | null` that selects which of 12 modules renders in the main pane. Because the review analysis ships embedded in the bundle (`src/data/reviewAnalysis.json` via `useAnalysis`), the normal runtime path is auth → apikey → straight to dashboard; the upload/processing branch only runs when the user explicitly uploads fresh review CSVs.

Once past the gates, everything renders inside `AppShell` (persistent left `Sidebar` + main pane). The sidebar is the navigation registry: an unlabeled Home row; a "Creative" section with two collapsible groups — "Briefing" (The Factory 🏭 as primary with Concepts & Angles, Script Writer, and Hooks nested as visually-subordinate subsections, plus Factory Anatomy as a peer explainer) and "Context Hub" (Customer Reviews, Ad Comments, Memory Vault, Inspiration); and an "Insights" section (Segments, Personas, Product Intel). `Home.tsx` is the dashboard landing page with live status cards (counts pulled from the comment bank, saved-analysis store, and inspiration store), a promo hero for the Factory Anatomy tour, and shortcut buttons.

Modules split into three archetypes. (1) **Single-shot generators** (SegmentDiscovery, PersonaBuilder, AnglesGenerator, HookGenerator, ScriptWriter) follow one pattern: a builder form of typed selectors → `buildXxxPrompt(params, analysis)` from `src/prompts/` → optional inspiration-context and marketing-brain addendum injection → `useClaudeApi.generate(system + resourceContext, user, tunedMaxTokens, model)` → a dedicated results view (`ResultsView`, `AnglesResultsView`, `PersonaResultsView`, `SegmentResultsView`) with copy/.md/.doc/.pdf export and feedback-driven regeneration through `FeedbackPanel` + `buildRegenerationPrompt`. (2) **Multi-phase pipelines** (AutopilotBriefs a.k.a. The Factory, CommentIntelligence) manage their own internal phase enums and call `sendMessage`/`sendVisionMessage` directly or through orchestrators in `src/autopilot/` and `src/engine/batchedCommentAnalysis`. (3) **Read-only dashboards** (CustomerReviews, ProductIntelligence, MemoryVault, FactoryTour) render deterministic engine/store data with zero AI calls.

The two hooks are the workhorses: `useAnalysis` owns the FullAnalysis + resourceContext (uploaded non-CSV files become a text block appended to every system prompt) and drives the CSV → organize → analyze progress pipeline entirely in-browser; `useClaudeApi` is a small abort-aware wrapper around `sendMessage` giving each module isolated result/loading/error state. A deliberate architectural theme throughout the results views is "hybrid rendering": deterministic charts computed from engine data never fail, while Claude's markdown narrative is parsed into structured cards with a plain-markdown fallback.

## Data flow

**Top-level state machine (App.tsx, `AppView` in engine/types.ts:144):** `auth → apikey → upload → processing → dashboard → module`. All state is React useState in App: `view`, `apiKey` (initialized from sessionStorage 'claude_api_key'), `activeModule: ModuleId | null`, `conceptAngleContext`. Transitions: PasswordGate onSuccess→'apikey'; ApiKeyInput onSubmit→'dashboard' IF `analysis` already exists (embedded data) else 'upload'; CsvUploader onFiles→'processing' + `processFiles()`; a useEffect flips 'processing'→'dashboard' 500ms after `progress.stage==='complete'`; a render-time derived `effectiveView` forces 'upload' if processing errored (no setState cascade). 'dashboard' and 'module' share one branch: both render inside `AppShell` (Sidebar + main pane); `activeModule===null` renders `Home`, otherwise a ternary chain maps each ModuleId to its component. Sidebar "Upload new data" → `handleReset` → view='upload' (analysis kept, so CsvUploader shows a "Back to dashboard" link).

**Cross-module handoff:** AnglesGenerator's `onWriteScript` bundles the selected concept markdown + product/awareness/funnel/adType/fullAi settings into a `ConceptContext`, App stores it and sets activeModule='script'; ScriptWriter then locks (disables) those selectors and injects the concept into the prompt. `handleBackToDashboard` clears it.

**useAnalysis (src/hooks/useAnalysis.ts):** owns `analysis: FullAnalysis`, `resourceContext: string`, `progress`, `error`. Initializes analysis from embedded JSON (`hasEmbeddedReviews()` → `embeddedReviewAnalysis`), so a fresh session normally never sees the upload screen. `processFiles(csvFiles, resourceFiles)`: PapaParse each CSV → merge RawReviews → read non-CSV files as text joined into `resourceContext` ("--- name ---\ntext") → `organizeReviews` → `runFullAnalysis` with progress remapped to 20–95% → setAnalysis. Everything runs in-browser; result replaces embedded data for the session (not persisted).

**useClaudeApi (src/hooks/useClaudeApi.ts):** per-instance `{result, loading, error, generate, reset, cancel}` around `sendMessage(system, user, apiKey, maxTokens=12000, model?, signal)`. `generate` aborts any in-flight request via AbortController ref; aborted requests resolve to null without touching state. Six modules use it (AnglesGenerator, HookGenerator, ScriptWriter, PersonaBuilder, SegmentDiscovery, CommentIntelligence-insights). Standard module pattern: builder form → `handleGenerate(feedback?)` builds `{system,user}` from the prompt builder, appends `buildResourceContext(resourceContext)` (+ optional brain addendum + inspiration context block), computes a hand-tuned maxTokens budget, calls `generate(...)`; when `result||loading||error` the module renders its results view instead of the form; "Regenerate with feedback" wraps the previous output + feedback via `buildRegenerationPrompt`.

**The Factory (AutopilotBriefs) phase machine (`BatchPhase`):** idle (chooser → screenshot upload OR ManualTaskBuilder) → parsing (`parseAsanaScreenshot` vision call) → confirming (PlannerView: include/exclude tasks, per-task ad-type override, pin inspiration, creative-direction instructions, MemoryPanel) → strategy-session (memory curator briefing + `runStrategySession` questions → user answers → `synthesizeStrategy` brief) → generating-concepts (`runConceptPhase`, PipelineProgress) → concept-review (ConceptReview; per-task concept regen with feedback re-runs angles+evaluator inline in the component) → running (`runScriptPhase`) → complete (BatchResultsView: TaskBriefCard + BriefPreviews + ScoreOverridePanel + BatchChatPanel + per-task redo via `redoSingleTask`) → error. Refs persist `directionRef` (CreativeDirection) and `memoryBriefingRef` across phases; a shared `abortRef` powers Cancel.

**CommentIntelligence phase machine:** list (saved analyses from IndexedDB; auto-opens the largest/combined analysis on first mount) → upload (CommentUploader: CSV parse OR Meta connect + CommentPullPanel 500-comment batch slices) → categorizing (`categorizeCommentsBatched`, 500/batch, concurrency 3, cancellable, partial-failure tolerant) → generating-insights (single aggregated call, 14000 tokens) → results (tabs: dashboard/feed/report). Every completed run is persisted as a `SavedAnalysis`; "Combine" merges ≥2 saved analyses, reruns insights once, saves as type 'combined'.

**Read-only dashboards (no Claude calls):** CustomerReviews, ProductIntelligence, FactoryTour, MemoryVault (localStorage memoryStore), Home (mounts → best-effort counts from commentBank, commentAnalysisStore, lazy-imported inspirationStore).

## Gotchas

- **Adding a module requires 3 edits**: `ModuleId` union (src/engine/types.ts:152), the ternary render chain in App.tsx (~line 116), and the `SECTIONS` registry in Sidebar.tsx (line 69). Home.tsx status cards / shortcuts are optional 4th. There is no router — no URLs, no deep links, browser back exits the app.
- **`src/components/OutputSelector.tsx` is dead code** — replaced by Home.tsx + Sidebar; nothing imports it.
- **Password/API gates are cosmetic-ish**: PasswordGate compares SHA-256 against `EXPECTED_HASH` in src/utils/hash.ts (plaintext is NOT "password" — ask the user). ApiKeyInput only validates the `sk-ant-` prefix and writes sessionStorage 'claude_api_key'; App reads it into state but the view still starts at 'auth' every reload.
- **Embedded data path**: `useAnalysis` boots with `embeddedReviewAnalysis` from src/data/reviewAnalysis.json, so 'upload'/'processing' are normally skipped; uploading new CSVs replaces the analysis for the session only (never persisted). `resourceContext` is only ever set via upload — in embedded mode it's '' and `buildResourceContext('')` must tolerate that.
- **Props are over-provided by design**: App spreads `moduleProps = {analysis, apiKey, resourceContext, onBack}` into most modules, but several ignore parts (CommentIntelligence destructures only `{apiKey, onBack}`; CustomerReviews/ProductIntelligence only `{analysis, onBack}`). InspirationBank gets only `{apiKey, onBack}`, MemoryVault only `{onBack}`, FactoryTour `{onBack, onOpenFactory}`.
- **Model routing is deliberate and mixed**: ideation calls pass `IDEATION_MODEL`, execution `CREATIVE_MODEL` (both 'claude-opus-4-8' today; IDEATION has a TODO to flip to gated 'claude-fable-5'), chat/parsing `UTILITY_MODEL`. SegmentDiscovery passes NO model, so it silently uses sendMessage's default 'claude-sonnet-4-6'. maxTokens budgets are hand-tuned per module/ad-type/duration (e.g. AGC ≤24000, personas ≤48000) — don't flatten them.
- **useClaudeApi holds ONE result per instance**; calling `generate` aborts the previous request. `cancel()`/`reset()` abort but note `setLoading(false)` is skipped in finally when aborted (reset/cancel set it explicitly). Regeneration is stateless prompt-stuffing: previous output + feedback via buildRegenerationPrompt — there's no conversation history except in PersonaChat/BatchChatPanel (sendChatMessage).
- **Persistence is fragmented across 4 backends**: sessionStorage (API key), localStorage (autopilot memoryStore under one key; customOptionsRegistry 'viasox_custom_options'), IndexedDB (inspirationStore, commentBank, commentAnalysisStore, VoC index), and nothing for FullAnalysis. MemoryStore APIs are synchronous; the IndexedDB stores are async.
- **Custom option lists**: AnglesGenerator/ScriptWriter build PRODUCTS/AD_TYPES/ANGLE_TYPES/FRAMEWORKS from `customOptionsRegistry` (`getAllProducts()` etc. memoized once per mount), but HookGenerator/PersonaBuilder hardcode the 3 product lines — inconsistent on purpose today.
- **Magic strings couple UI to prompts**: AnglesResultsView splits concepts on the literal `## Concept N:` heading the prompt enforces; BriefPreviews/downloadUtils parse markdown tables with `normalizeBodyRow` so preview and .doc/.csv export always match — change brief templates and both sides must move together. `FULL_AI_AD_TYPE` is the exact string 'Full AI (Documentary, story, education, etc)'.
- **Timeouts are huge**: api/claude.ts gives Opus calls a fixed 30-min timeout, Sonnet 15-min; proxy first, direct API fallback.
- **Theme tokens are custom Tailwind v4 CSS variables** in src/index.css (`--color-cream*`, `--color-navy*`, `--color-warm-*`, `--font-display` = Newsreader serif). Classes like `bg-cream`, `text-navy`, `font-display`, `hover-lift`, `animate-fade-in` only exist because of that file — Tailwind v4 via `@tailwindcss/vite`, no postcss/tailwind.config.
- **StrictMode is on** (main.tsx) — mount effects run twice in dev; Home's count loading and CommentIntelligence's auto-open guard (`hasAutoLoadedRef`) already account for this.
- **Home's "View all" links are intentional no-ops** (`e.preventDefault(); void viewAllTarget`) and three sections are placeholders awaiting content. SegmentDiscovery's yearly-YOY math hardcodes exclusion of partial year `'2026'`.
- **AutopilotBriefs.handleRegenerateConcepts duplicates pipeline logic inline** (angles prompt + vision frames + evaluator + pinned-framework forcing) rather than calling pipelineEngine — keep the two in sync when changing concept-phase behavior.

# 2. Analysis Engine & Comment Intelligence

**Scope:** `src/engine/`, `src/comments/`

## Overview

These two folders form the app's "data intelligence" layer — everything that turns raw customer voice (Shopify product reviews and Meta ad comments) into structured, deterministic data that the Claude prompt builders can enrich. There are two independent pipelines that never call each other.

**Pipeline 1 — Review analysis (src/engine/)**: pure, deterministic, zero-AI regex analysis of product review CSVs. `organizeReviews.ts` buckets reviews into 4 product categories by handle substring; `analyzeReviews.ts` runs per-product pattern counting (pain/benefit/transformation/segment regexes from `patterns.ts`) plus transformation-story extraction; `segmentEngine.ts` builds the two-layer segment model (Layer 1 identity = WHO: healthcare worker, diabetic, senior, etc.; Layer 2 motivation = WHY: comfort seeker, pain relief, skeptic converted, etc.), including identity×motivation cross-overlaps and per-product concentration indexes. The design principle is stated in the code: "same input always produces same output. No AI involved. Claude only enriches these segments later." The output (`FullAnalysis`) feeds every module and is also serialized into `formatSegmentBreakdown()` prompt text. In practice the app usually skips this pipeline at runtime: `src/data/reviewAnalysis.json` embeds a pre-computed FullAnalysis from 110K+ reviews (built by `scripts/preprocess-reviews.mjs`, which *duplicates* the engine logic in plain JS), and `useAnalysis.ts` initializes from it — CSV upload only overrides it.

**Pipeline 2 — Comment intelligence (src/comments/ + engine/batchedCommentAnalysis.ts)**: AI-driven analysis of Facebook ad comments. `commentPuller.ts` walks the Meta Graph API through the read-only Cloudflare Worker proxy (`src/api/metaProxy.ts`) — refresh page tokens → discover ad accounts → discover ads with post creatives → pull comments per ad with 90-day backfill or delta cursors — and stores everything in the IndexedDB "Comment Bank" (`commentBank.ts`, DB `viasox_comments`). The UI slices the bank into 500-comment batches (newest-first); `batchedCommentAnalysis.ts` categorizes each batch through Claude (category/sentiment/theme per comment) with a 3-worker promise pool and per-batch retry, then a single Phase-2 insights call generates a markdown report. Completed runs are persisted as `SavedAnalysis` snapshots in a second IndexedDB (`commentAnalysisStore.ts`, DB `viasox_comment_analyses`), which the Brain's VoC index (`src/brain/vocIndex.ts`) also reads.

The remaining two engine files are pure type modules: `autopilotTypes.ts` (Asana-driven brief batch pipeline: task parsing → strategist → concepts → critique → selection → script) and `inspirationTypes.ts` (Inspiration Bank: analyzed reference ads with tags, learnings, and contextual performance scores keyed "angleType|duration"). They live in engine/ only as shared type homes; their logic is in src/autopilot/ and src/inspiration/.

## Key files

- `src/engine/types.ts` — central type module (57 importers): `RawReview`, `ProductCategory` (4 fixed values), `FullAnalysis`, `SegmentProfile`/`SegmentBreakdown`, `CrossSegmentOverlap`, `ProductAffinityMap`, `YearlyTrends`, plus app-wide types (`AppView`, `ModuleId`, all module param interfaces: `PersonaParams`, `AnglesParams`, `HooksParams`, `ScriptParams`, and the `AdType`/`AngleType`/`HookStyle`/`ScriptFramework` string-literal unions) and Claude API request/response shapes.
- `src/engine/patterns.ts` — all regexes: `PAIN_PATTERNS` (10), `BENEFIT_PATTERNS` (10), `TRANSFORMATION_PATTERNS` (11), `IDENTITY_SEGMENT_PATTERNS` (9), `MOTIVATION_SEGMENT_PATTERNS` (8), merged `SEGMENT_PATTERNS`, `TRANSFORMATION_STORY_PATTERN`. All case-insensitive, non-global.
- `src/engine/organizeReviews.ts` — `categorizeProduct(handle)` buckets by substring ('easystretch' → EasyStretch; 'ankle-compression' checked BEFORE 'compression'; else Other) and `organizeReviews()` groups a RawReview[] into `OrganizedReviews`.
- `src/engine/analyzeReviews.ts` — `runFullAnalysis(organized, onProgress)`: per-product stats (avg rating, 5★/1★ %), pattern counts with up to 5 quotes each (min 50 chars, truncated at 300), up to 20 transformation stories (min 100 chars, truncated at 500), then calls `buildSegmentBreakdown`. Yields to UI via requestAnimationFrame between products. Skips the 'Other' bucket for per-product analysis.
- `src/engine/segmentEngine.ts` — `buildSegmentBreakdown(organized)`: tags every review with all matching segments, builds per-segment profiles (byProduct split, top-5 pains/benefits/transformations, ratings, quotes preferring 5★ then longest), unsegmented/multiSegment counts, `computeCrossSegmentOverlaps` (identity×motivation pairs sorted by count), `computeProductAffinity` (share-of-product + concentration index). Also `formatSegmentBreakdown(breakdown, maxSegments=12, maxQuotes=2)` — renders the breakdown as prompt markdown, motivation segments first, quotes trimmed to 180 chars.
- `src/engine/batchedCommentAnalysis.ts` — `categorizeCommentsBatched(comments, apiKey, options)`: chunks into 500-comment batches, 3-concurrency promise pool, each batch retried once with 1–3s jitter, local→global index translation (`i + offset`), out-of-range indices dropped, failed batches logged and skipped (their comments fall through to `{cat:'Engagement', sent:'Neutral', theme:''}` defaults in the caller). Parses markdown-fenced JSON defensively.
- `src/engine/autopilotTypes.ts` — types only: `ParsedAsanaTask`, `AutopilotTask`, `PipelineStep`/`BatchPhase` state machines, `TaskPipelineState`, `ConceptOption`, `CreativeDirection`, `StrategySession`. Used by src/autopilot/* and components/autopilot/*.
- `src/engine/inspirationTypes.ts` — types + one helper `getEffectiveTags(item)` (user overrides spread over analyzer tags). `InspirationItem` includes performance-feedback fields (`usageCount`, `derivedScore`, `contextualScores` keyed "angleType|duration"). Used by src/inspiration/*.
- `src/comments/commentBankTypes.ts` — `CommentRecord` (id = `${adId}:${commentId}` composite key), `AdSyncCursor`, `CommentBankStats`, `PullProgress`, `PullSummary`.
- `src/comments/commentBank.ts` — IndexedDB `viasox_comments` v1, stores `comments` (indexes: adId, campaignId, createdAt, pulledAt), `cursors`, `meta`. `putComments` uses put-semantics and returns {added, updated}; `clearAllComments` wipes all three stores.
- `src/comments/commentPuller.ts` — `pullAllAdComments(onProgress, signal)`: the full Meta pull orchestration (see data flow). Constants: `BACKFILL_DAYS=90`, page sizes 100.
- `src/comments/commentAnalysisStore.ts` — IndexedDB `viasox_comment_analyses` v2, store `analyses`: `SavedAnalysis` snapshots (full categorizedComments + summary + insightsReport markdown). Helpers: `newAnalysisId`, `deriveDateRange`, `mergeAnalysesComments` (dedupes by commenter|text|date). save/delete invalidate the brain's VoC cache via lazy import.

## Data flow

**Review pipeline (deterministic, browser-only, no network):** CSV file(s) → `parseCsv` (src/utils/csv.ts, PapaParse) → `RawReview[]` → `organizeReviews()` buckets by handle substring → `runFullAnalysis(organized, onProgress)` → per-product `ProductAnalysis` (pattern counts + %s + quotes + transformation stories) + `buildSegmentBreakdown()` (tag → profile → cross-overlaps → product affinity) → `FullAnalysis` held in `useAnalysis` React state and passed as a prop into every module. Progress flows via the `ProcessingProgress` callback ({stage, currentProduct, percent}); useAnalysis remaps engine 0–100% into display 20–95%. Default path: `useAnalysis` initializes state from `embeddedReviewAnalysis` (static JSON import) so the upload step is optional. The breakdown reaches Claude only as formatted text via `formatSegmentBreakdown()` (used by src/prompts/segmentsPrompt.ts); `systemBase.ts` also embeds the segment pattern *names* into the base system prompt.

**Comment pull (Meta → IndexedDB):** `CommentPullPanel` → `pullAllAdComments()`: (0) `refreshPageTokens()` on the Worker — hard-fails with diagnostic text if 0 page tokens cached (page-post comment endpoints require page tokens, not user tokens); (1) paginate `me/adaccounts`; (2) per account paginate `{act}/ads` keeping only ads with `creative.effective_object_story_id` and effective_status ACTIVE/PAUSED/IN_PROCESS; (3) per ad, read `AdSyncCursor` — no cursor ⇒ 90-day backfill, else `since = newestCommentTime`; extract page_id from the postId prefix (`{page_id}_{post_id}`); paginate `{postId}/comments` (order=chronological, fields id,message,created_time,from,like_count) through `metaGraph({page_id})`; convert to `CommentRecord` (skip empty messages, dedupe within batch); `putComments` (put-semantics, counts new); update cursor; per-ad errors are caught and recorded in `PullSummary.perAd`, never fatal. Finish: `setLastPullAt`. Pagination follows Meta's `paging.next` URL but re-routes it through the Worker after stripping any access_token param.

**Comment analysis (bank/CSV → Claude → saved snapshot):** Entry points: (a) CSV upload → `parseCommentCsv` (column-alias detection; requires a comment column; caps at MAX_COMMENTS=10,000) or (b) bank batch — CommentPullPanel sorts the bank newest-first, slices 500-comment window `batchIndex` (1-indexed), maps `CommentRecord → RawComment` via `commentRecordToRaw`. Both call `CommentIntelligence.handleCommentsReady(comments, source)`. Phase 1: `categorizeCommentsBatched` → `buildCommentCategorizationPrompt` per 500-batch → `sendMessage` (24K max tokens/batch) → JSON `{results:[{i,cat,sent,theme}]}` → global index merge; caller maps indices back onto originals, defaulting missing ones to Engagement/Neutral. Phase 2: `buildSummary` (byCat/bySentiment/topThemes) + 5 samples per category → `buildCommentInsightsPrompt` → ONE Claude call (14K tokens) → markdown report. Result is persisted as `SavedAnalysis` (type 'batch' | 'csv') and rendered in dashboard/feed/report tabs. Combine flow: merge ≥2 saved analyses (`mergeAnalysesComments` dedup) → rebuild summary → one fresh insights call → save as type 'combined'. On module mount, the largest analysis (preferring 'combined') auto-opens. `saveAnalysis`/`deleteAnalysis` invalidate the brain's VoC index, which rebuilds lazily from `getAllAnalyses()` + the review FullAnalysis.

**State:** review analysis is in-memory React state (plus build-time embedded JSON); comment bank + cursors + saved analyses + VoC cache are IndexedDB (survive reloads, device-local); API key is sessionStorage (outside this subsystem).

## Gotchas

1. **Triplicated pattern/engine logic:** `scripts/preprocess-reviews.mjs` re-implements patterns.ts + analyzeReviews.ts + segmentEngine.ts in plain JS to generate the embedded `src/data/reviewAnalysis.json`. Editing a regex in patterns.ts does NOT change what users see unless the preprocessor is re-run (and its copied patterns updated) — the app boots from the embedded JSON, not runtime analysis.
2. **`yearlyTrends` is preprocessor-only:** `FullAnalysis.yearlyTrends` is consumed by SegmentDiscovery/CustomerReviews but is only produced by preprocess-reviews.mjs. The in-browser `runFullAnalysis` never populates it, so uploading a fresh CSV silently loses YOY trends.
3. **IndexedDB version coupling:** `commentAnalysisStore.ts` and `src/brain/vocIndex.ts` open the SAME database (`viasox_comment_analyses`) and their `DB_VERSION` constants must stay in sync (both currently 2). A mismatch throws VersionError, which is silently caught — the UI just shows an empty analyses list while data still exists. The comment there is explicit; keep both bumped together.
4. **Product categorization order matters:** in `categorizeProduct`, 'ankle-compression' must be checked before 'compression' (substring shadowing). Product categories are hardcoded to exactly 4 values; the 'Other' bucket is counted but never analyzed per-product or segmented.
5. **Failed-batch semantics:** comments in failed categorization batches silently become `{category:'Engagement', sentiment:'Neutral', theme:''}` in the final data — they look categorized but aren't. `unCategorizedComments` in the result is an estimate (`failedBatches * batchSize`), not exact for the last short batch. A non-fatal banner is the only signal.
6. **Batch sizing constants live in two places:** engine default `DEFAULT_BATCH_SIZE=500` (batchedCommentAnalysis.ts) and UI `BATCH_SIZE=500` (CommentPullPanel.tsx slicing). They coincide but are independent; `DEFAULT_MAX_TOKENS_PER_BATCH=24_000` is sized for 500 comments × ~40 output tokens — change batch size and token budget together.
7. **Comment identity is composite:** `CommentRecord.id = ${adId}:${commentId}` because Meta surfaces the same comment against multiple ads (carousels). Cross-ad duplicates therefore exist in the bank by design; `mergeAnalysesComments` dedupes at analysis level by (commenterName|text|date) instead.
8. **Delta cursor subtlety:** `AdSyncCursor.newestCommentTime` drives the `since=` param (unix seconds). Cursors update even on zero-comment pulls; the whole delta scheme assumes `order=chronological` pagination. Deleting cursors (only via clearAllComments) re-triggers the 90-day backfill.
9. **Page tokens are mandatory for comments:** Meta rejects user tokens on `/POST_ID/comments`. `pullAllAdComments` hard-fails before walking ads if `refreshPageTokens()` caches zero page tokens, with a long diagnostic message telling the user to open "Diagnose page access". `extractPageId` parses `{page_id}_{post_id}`; Instagram-only/Dynamic Creative post IDs don't match and fall back to the user token (likely failing per-ad, caught non-fatally).
10. **Regexes must stay non-global:** all pattern regexes use bare `/i`. Adding a `g` flag would break `.test()` due to stateful `lastIndex` across the thousands of sequential tests.
11. **Determinism is a design contract:** segmentEngine/analyzeReviews are intentionally AI-free ("Claude only enriches these segments later"). Don't add Claude calls into the engine; enrichment belongs in src/prompts/*.
12. **Percentage rounding is inconsistent by intent:** analyzeReviews rounds to 2 decimals; segmentEngine to 1 decimal (`*1000/10`). Snapshot-style comparisons must account for this.
13. **Claude call behavior:** `sendMessage` tries the CORS proxy first, then direct api.anthropic.com with `anthropic-dangerous-direct-browser-access`; retries 429/529 up to 8 times (15s→180s backoff); Opus gets a fixed 30-min timeout, everything else 15 min; truncated (`stop_reason==='max_tokens'`) responses get a visible note appended to the returned text — downstream JSON parsing of a truncated categorization batch will throw and consume that batch's single retry.
14. **CSV comment cap:** CommentUploader truncates to 10,000 comments (MAX_COMMENTS) — a soft safety ceiling, since batching removed the old single-prompt limit.
15. **`inspirationTypes.userCategories` is UI-only** (folder tabs); the analyzer/selector/injection logic deliberately ignore it. Contextual scores are keyed by the literal string `"angleType|duration"`.

# 3. Prompt System

**Scope:** `src/prompts/`

## Overview

`src/prompts/` is the prompt-engineering core of the Viasox Marketing Intelligence SPA — ~10,860 lines of TypeScript whose sole job is assembling `{ system, user }` prompt pairs for Claude API calls made from the browser. There is no runtime logic here beyond string composition: every file either (a) exports a `build*Prompt(params, analysis, …)` function for one pipeline step, or (b) exports reusable knowledge blocks (manifesto excerpts, marketing-book distillations, awareness-level guides, claim boundaries, output templates) that the step prompts compose. The directory encodes the entire Viasox brand brain in code: canonical brand facts (12-15 mmHg compression, EasyStretch stretches 30", B2G3 = 5 pairs/$60, 107,993 reviews, "women 50+" audience mandate), the Feb/Mar/Apr 2026 Marketing Manifesto updates, Schwartz's five awareness levels (with an April-2026 policy that Unaware is the DEFAULT TOF level and Problem-Aware is reclassified Upper-MOF), 20 script frameworks, angle language banks, objection banks, a "winning ad reference bank" of studied top performers, and a full Marketing Brain (7 markdown distillations of Schwartz/Hopkins/Bly/Neumeier + 2 video masterclasses, imported via Vite `?raw` from `src/knowledge/brain/`).

Two prompt families exist. The **standalone module prompts** power the interactive dashboard modules: `reportPrompt` (Review Intelligence Report), `anglesPrompt` (Concepts & Angles generator), `hooksPrompt` (Hook generator, script-tailored or standalone mode), `scriptPrompt` (Script/Brief writer), `personaPrompt` (Persona Builder for DTC/Retail/Wholesale channels), `segmentsPrompt` (Segment Discovery narrative over deterministic regex segmentation), and `commentPrompt` (ad-comment categorization + insights). The **autopilot agent prompts** power the multi-agent weekly-brief pipeline in `src/autopilot/pipelineEngine.ts`: `strategySessionPrompt` (batch analysis + Q&A synthesis into a Weekly Strategy Brief), `creativeStrategistPrompt` (per-brief creative thesis before generation), `anglesPrompt` again (concept generation), `differentiationCriticPrompt` (relevance + claim-grounding gate, can trigger one regeneration), `conceptSelectorPrompt` (Opus-grade single-winner selection) / `conceptEvaluatorPrompt` (rate-all-concepts variant for the interactive review UI), `scriptPrompt` (brief writing per ad-type template), and `batchReviewerPrompt` (strict QC: 11 per-brief criteria scored 1-10 as JSON + 4 batch checks + systemic pattern analysis). `regenerationPrompt` wraps any original user prompt with previous output + user feedback for directed re-rolls.

The architecture's central discipline is a **hierarchy of authority** repeated in the generator/selector/writer prompts: (1) user directives (talking point, angle type, duration, pinned inspiration) > (2) awareness-level rules > (3) ad-type format rules > (4) funnel-stage pacing > (5) manifesto background > (6) general marketing principles. This exists because the manifesto blocks are voluminous and historically pulled the model toward generic "manifesto-flavored" concepts; several agents (Creative Strategist, Differentiation Critic) exist specifically to counteract that drift. A second discipline is the **claim boundary** (`claimBoundary.ts`), added after a June 2026 audit found the pipeline inventing pains/benefits (worst on Ankle Compression, whose recorded claim space is ~7 pains): diversity is defined as execution-level (persona/moment/format), never claim-level, and the critic auto-rejects any concept whose central claim can't be traced to recorded triggers, review data, or the assigned talking point.

## Data flow

Every builder is a pure function returning `{ system: string, user: string }` (regenerationPrompt returns just a user string; commentPrompt's categorizer returns a JSON-contract pair). Inputs come from three places: (1) typed params from `src/engine/types` (`AnglesParams`, `ScriptParams`, `HooksParams`, `PersonaParams`, `AutopilotTask`, enums for `ProductCategory`, `AdType`, `AwarenessLevel`, `AngleType`, `FunnelStage`, `FullAiSpecification/VisualStyle`); (2) the `FullAnalysis` object produced by the in-browser review-analysis engine, formatted into prompt text by systemBase helpers (and by `personaPrompt` via `segmentEngine`/`salesEnrichmentLoader`/`marketIntelligence`); (3) optional pipeline context strings threaded by callers — `memoryBriefing` (institutional memory curator), `inspirationContext` (inspiration bank), `anglePatternTable` (mined performance history), `strategyBrief`, `conceptAngleContext`, `scriptContext`, `calibrationBlock`.

Composition pattern: each step prompt string-interpolates `buildSystemBase()` first, then a "TOP-PRIORITY USER DIRECTIVES" block with the hierarchy of authority, then per-selector guides (ad type, funnel, awareness, product, angle mandate, claim boundary), then a long tail of manifesto getters, and finally appends memory/inspiration sections and `getMarketingBrainBlock(step)` (the brain block is appended LAST to the system string in every consumer). The manifesto data is static and compiled into the JS bundle — nothing is fetched at runtime; the brain `.md` files are inlined at build time by Vite `?raw` imports.

Autopilot flow (driven by `src/autopilot/pipelineEngine.ts`): strategy analysis → user answers questions → strategy synthesis (Weekly Strategy Brief) → per task: creative strategist thesis → anglesPrompt (5 concepts, with thesis/memory/inspiration injected by caller) → differentiation critic (KEEP/REJECT; may trigger ONE regeneration) → concept selector (or concept evaluator when the user reviews interactively) → scriptPrompt with `conceptAngleContext` (concept overrides all other selectors) → batch reviewer QC. Several prompts demand structured outputs that co-located parsers extract: XML (`parseStrategyAnalysis`, `parseConceptEvaluations`), labeled fields (`parseSelectorResponse`), markdown sections (`parseDifferentiationCritique`), JSON fences (batch reviewer, comment categorizer). Prompt text and parser regexes are therefore tightly coupled — changing an output-format instruction requires changing the sibling parser.

Two mandatory "think before you write" scaffolds shape generation quality: anglesPrompt's Step-0 Territory Map (5 distinct territories, unique real proof anchors, hook-archetype distribution, claim-boundary self-check) and scriptPrompt's Step-0 Strategic Outline (beat map, word budget vs hard ceiling, talking-point threading in ≥3 beats, hook archetype uniqueness).

# 4. Marketing Brain & Data Loaders

**Scope:** `src/brain/`, `src/data/`

## Overview

**The Marketing Brain (`src/brain/`)** is an opt-in context-assembly layer that enriches the app's existing Claude prompt builders with grounded customer-voice evidence. It never replaces a prompt — it produces an *addendum* string that callers append verbatim to their existing system prompt (`system + brain.addendum`). Its core knowledge source is a pre-computed **Voice-of-Audience (VoC) index**: a structured aggregation of (a) the current session's review `FullAnalysis` (110K+ reviews) and (b) all saved comment analyses persisted in IndexedDB by the comments subsystem. The index groups themes into slices — topObjections, topTestimonials, recurringQuestions, painPoints, desires, complaints, personaSignals (beth/linda/other), productSignals (compression/easystretch/ankle), emergingThemes — each a list of `VoCItem`s (theme + frequency + up to 5 verbatim quotes + sentiment/category). Index building is pure TypeScript (no LLM calls), cached in IndexedDB, and lazily rebuilt when invalidated.

The single public entry point is `buildBrainAddendum(task, options)` in `contextAssembler.ts`. Given a `BrainTaskDescriptor` (module, product, persona, template, angle, batch info), it: (1) checks the per-module feature flag (off → strict no-op empty addendum), (2) ensures the VoC index is fresh, (3) selects task-relevant slices via `sliceSelector.ts` (conservative: objections + testimonials always; more slices for strategists/producers), (4) optionally fires a second "deep reasoning" Claude call (Sonnet, `UTILITY_MODEL`, max 2000 tokens, ~$0.04, 10–25s) that produces a gap analysis over the selected slices, and (5) renders everything to markdown via `addendumRenderer.ts` — a `\n\n---\n\n`-delimited block with labeled headers (`## VOICE OF AUDIENCE — RECENT DATA`, `## DEEP REASONING — GAP ANALYSIS FOR THIS TASK`). Every internal failure degrades gracefully (empty addendum or addendum without deep block + warning); the caller's main generation is never blocked. Deep reasoning fires only for: batch ≥ 3, `aidoc` template, `strategySession` or `creativeStrategist` modules, or `forceDeepReasoning`. Within an autopilot batch, `brainSession.ts` caches deep-reasoning output under a session ID so N briefs cost 1 deep-reasoning call, not N.

**The data loaders (`src/data/`)** are a separate, complementary knowledge source: build-time static intelligence (as opposed to the brain's runtime-computed VoC index). `reviewAnalysisLoader.ts` statically imports `reviewAnalysis.json` (232K, 110,347 reviews pre-analyzed offline by `scripts/preprocess-reviews.mjs` — identical structure to the in-browser `runFullAnalysis()` output), so the app has full review analysis with no CSV upload. `salesEnrichmentLoader.ts` statically imports `salesEnrichment.json` (476K, generated by `scripts/preprocess-sales-data.mjs` from 2.5M order lines / 619K customers / 110K reviews, reviewer↔order email linking at 89% link rate) with 17 named segments carrying full sales economics (LTV, repeat rate, cross-purchase, geography). `marketIntelligence.ts` is hand-curated, embedded secondary market research (US/CA market sizes, retailer profiles, segment benchmarks, scoring frameworks — last updated Feb 2026) exposed both as structured objects for UI and as prompt-ready markdown strings via `buildMarketContext()`.

## Data flow

**Brain flow:** Caller builds a `BrainTaskDescriptor` and awaits `buildBrainAddendum(task, {reviews, apiKey, sessionId})`. Steps: (1) `isBrainEnabledFor(module)` — if off, immediate `{addendum: '', metadata}` no-op. (2) `ensureFreshVoCIndex(reviews)` — returns the IndexedDB-cached index if present, else rebuilds: pulls all saved comment analyses via `getAllAnalyses()` (from `src/comments/commentAnalysisStore.ts`) + walks the passed review `FullAnalysis` (products' pain/benefits/transformation/segments categories) and each `CategorizedComment` (routed by category: Objection/Testimonial/Question/Request/Complaint/Engagement; Spam skipped), deduping themes by lowercase handle, merging review+comment sources into `source:'both'`, capping 20 items/slice and 5 quotes/item. Note: in v1, review data does NOT feed product or persona slices (stub functions return null) — those are comment-derived only via regex keyword sniffing. (3) `selectSlicesForTask` picks a focused subset per module. (4) If `shouldRunDeepReasoning(task)` and a `sessionId` cache miss: `runDeepReasoning` sends the task descriptor + selected slices as JSON to Sonnet and gets back a ≤1200-word markdown gap analysis; result is cached under the sessionId. No apiKey → skip with warning. (5) Renderer produces the final addendum. Caller concatenates: e.g. `ScriptWriter.tsx` line 218: `generate(system + buildResourceContext(resourceContext) + brain.addendum, finalUser, maxTokens, CREATIVE_MODEL)`.

**Invalidation:** `commentAnalysisStore.saveAnalysis()`/`deleteAnalysis()` lazily import and call `invalidateVoCIndex()` (lazy import avoids a circular dependency), so the next brain call rebuilds. There is no time-based staleness — only explicit invalidation.

**Autopilot session flow:** `pipelineEngine.ts` calls `endCurrentBrainSession()` at batch start (line 525) and end (line 1497), and threads `ensureCurrentBrainSession()` as `sessionId` into every brain call across the batch (strategySession ×2, conceptEvaluator ×2, briefGenerator ×3, conceptSelector ×1 call sites; `creativeStrategist.ts` and `differentiationCritic.ts` do the same). The first deep-reasoning fire (typically strategySession) populates the cache; all later calls in the batch hit it (metadata.deepReasoningCacheHit).

**Data loaders flow:** all three are synchronous build-time imports — no fetch, no upload. `useAnalysis.ts` seeds app state with `embeddedReviewAnalysis` when `hasEmbeddedReviews()`, which is also what gets passed as `options.reviews` into brain calls. `personaPrompt.ts` injects `salesEnrichment` numbers and `buildMarketContext(channel === 'Retail', markets)` markdown directly into the Persona Builder system prompt. Persona UI components (`src/components/persona/*`) read `salesEnrichment` structured fields directly for deterministic charts (Venn, radar, geography, cross-purchase), and `MarketGeoComparison.tsx` renders `US_MARKET_DATA`/`CA_MARKET_DATA`.

# 5. Inspiration Bank

**Scope:** `src/inspiration/`

## Overview

The `src/inspiration/` subsystem implements the **Inspiration Bank** — a browser-local library of reference ads that the app's generation pipelines use to make Claude mirror proven creative work instead of inventing generic patterns. Users upload two kinds of items (`InspirationKind = 'video' | 'brief'`): video ad files (mp4 etc.) and written briefs/scripts (.txt/.md/.docx/.pdf). There are no standalone image items — "briefs" and "scripts" were collapsed into a single `'brief'` kind (legacy `'script'` items are normalized to `'brief'` on read). Everything persists in IndexedDB (`viasox_inspiration` DB), so the bank is per-browser, never server-side.

Ingestion → analysis: because Claude vision can't ingest video, `frameExtractor.ts` samples up to 16 (default 8) evenly-spaced JPEG frames via `<video>` + Canvas (max 768px wide, quality 0.78, skipping the first/last 2% to avoid black title cards) plus a 320px thumbnail; `textExtractor.ts` extracts plain text from documents with lazy-loaded mammoth/pdfjs. `inspirationAnalyzer.ts` then sends frames (as base64 image content blocks) or text plus the prompt from `inspirationAnalyzerPrompt.ts` to Claude (`CREATIVE_MODEL` = claude-opus-4-8, 8000 max tokens) acting as an "expert Viasox ad strategist". The analyzer returns strict JSON: a fixed tag taxonomy (duration/adType/angleType/framework/hookStyle/isFullAi/fullAiSpecification/fullAiVisualStyle/productCategory/emotionalEntry/customTags — exact enum strings mirrored from `src/engine/types.ts`), plus rich creative fields: summary, learnings[], styleNotes, visualBlueprint, hookBreakdown (first 3 seconds), narrativeArc, productBridge, keyLanguage, lineFlowAnalysis, and a suggestedTitle used to auto-name filename-titled items (a separate cheap `UTILITY_MODEL` = claude-sonnet-4-6 pass backfills names for old items).

Selection → injection is **two-tier**: (1) **Pinned** — the user explicitly pins one bank item to a specific autopilot task (`CreativeDirection.pinnedInspirations`, a `Record<taskName, itemId>` built in `ManualTaskBuilder.tsx`/`PlannerView.tsx`); `pipelineEngine.loadPinnedInspirationForTask` loads its full frames (capped at 8 for vision calls) and builds a "PINNED REFERENCE — FOLLOW CLOSELY" block that also **locks the framework and hook style** of the generated script to the pin's tags. (2) **Matched** — when no pin exists, `inspirationSelector.selectInspiration` scores every ready item against the task context (hard filters: isFullAi boundary is always hard; adType is hard except a "short-form punch-through" where any 1-15 sec reference qualifies for a 1-15 sec brief; additive scoring: +5 adType, +4 angle, +3 framework, +3 product, +2/+6 duration with short-form priority and a -2 penalty for non-short-form refs on short-form briefs, +2 starred, +1 recency ≤30 days, plus a closed-loop performance boost of -3..+6 derived from past review scores, preferring contextual `angleType|duration` scores over the flat average; min score 3). `inspirationInjection.ts` renders the picks as markdown context: `buildInspirationContext` (light, used by standalone AnglesGenerator/ScriptWriter, default 5 picks) or `buildDeepInspirationContext` (autopilot, up to 10 picks, top 5 as full "mirror this exactly" blueprints including the actual reference script text, the rest as a lighter "breadth tier"; the top pick is PRIMARY and gets 4 of its frames attached as vision images).

The system closes a **learning loop**: pipeline runs record which item IDs were injected per brief (`inspirationIdsUsed`); after batch review, `src/autopilot/memoryExtractor.ts` calls `recordInspirationUsage` to update each item's flat rolling-average `derivedScore`, contextual `(angleType, duration)` scores with per-criterion averages, usage counts, and batch history — then `autoStarHighPerformingItems` auto-stars items with derivedScore ≥ 8.0 (n ≥ 3) and auto-unstars only auto-starred items that fall below 6.5. Manual stars are never removed. This makes winners rise and losers sink in future selections.

Ingest (UI: `src/components/modules/InspirationBank.tsx`): user drops a file + optionally sets two explicit tags (ad type, short-form checkbox → seeded as `userTagOverrides`) → `generateId()` (`insp_<timestamp>_<rand>`) → for video: `extractVideoFrames()` → `putItem(item, blob, frames)` with `status:'analyzing'` → `analyzeVideoItem()` (frames as vision image blocks + prompt); for documents: `extractTextFromFile()` → `putItem(item, blob)` → `analyzeTextItem()`. Selection: callers build `SelectionOptions` (adType, angleType, productCategory, duration, isFullAi, fullAiSpec, fullAiVisualStyle, framework, hookStyle, maxResults, starredOnly) → `selectInspiration()` loads ALL items from IDB, applies hard filters, scores, sorts, caps → `ScoredInspiration[]` with human-readable `matchReasons`.

Injection paths (three): (1) Standalone modules `AnglesGenerator.tsx` / `ScriptWriter.tsx` call `getInspirationContextBlock()` (light block, text-only) and pass it into their prompt builders. (2) Autopilot (`src/autopilot/pipelineEngine.ts`, also `AutopilotBriefs.tsx` redo path): per task, first tries `loadPinnedInspirationForTask(taskName, direction.pinnedInspirations)` — if a pin exists it wins outright (`inspirationMode:'pinned'`), all 8-capped frames attach via `buildPinnedVisionContent`, and framework/hookStyle locks propagate into concept selection; otherwise `loadDeepInspirationForTask(task)` (`inspirationMode:'matched'`, maxResults 10, primary pick's 4 frames attach via `buildDeepInspirationVisionContent`); else `'none'`. The same context block is reused across the task's stages (strategist → angles → concept selection → script). (3) `getAllItems()` also feeds the memory curator (`runMemoryCurator`) a compact list of the whole bank with scores, and `Home.tsx` lazily imports the store just for a count.

Feedback: pipeline stamps `ts.inspirationIdsUsed` (pin id and/or all deep pick ids, deduped via `mergeInspirationIds`) → after batch review, `memoryExtractor.ts` iterates each scored brief and calls `recordInspirationUsage(id, finalScore, batchId, {angleType, duration, criterionScores})` sequentially (to avoid IDB races), then `autoStarHighPerformingItems()`. State: everything is in IndexedDB; there is no React state store — components read/write the store directly.

## Gotchas

- **IndexedDB only, browser-local**: DB `viasox_inspiration` v1. Bank contents don't sync across machines/browsers; `exportMetadata()` exports metadata JSON only — blobs/frames are NOT exported, so there is no full backup/restore path.
- **Tag strings are load-bearing magic strings**: the analyzer prompt's TAG_TAXONOMY, `src/engine/types.ts` enums, selector comparisons, and hard filters all rely on exact string equality (e.g. `'Full AI (Documentary, story, education, etc)'`, `'1-15 sec'`). `pipelineEngine.loadDeepInspirationForTask` derives `isFullAi` by comparing adType to that exact Full-AI string. Changing any enum value breaks selection silently (items just stop matching).
- **`getEffectiveTags()` everywhere except pins**: user tag overrides layer over analyzer tags via `getEffectiveTags`. The selector, filters, stats, and injection all use it — but `loadPinnedInspirationForTask` in pipelineEngine reads `item.tags` directly, so user overrides do NOT affect pinned framework/hook locks.
- **Two-tier precedence**: a pin completely replaces matched selection (`deep` is not loaded when pinned). Pins also LOCK framework and hookStyle for the whole generated brief ("NON-NEGOTIABLE" prompt language + `opt.recommendedFramework` overwrite in AutopilotBriefs).
- **Short-form (`'1-15 sec'`) is special-cased in three places**: selector (punch-through past the adType hard filter, +6 duration bonus, -2 penalty for non-short-form refs), analyzer prompt (SHORT_FORM_LENS), and the visualBlueprint field contract. The Full-AI boundary, however, is never crossed — even by short-form punch-through.
- **Legacy normalization on read**: `normalizeKind()` maps persisted `kind:'script'` → `'brief'` and backfills `usageCount/derivedScore/derivedScoreSampleSize/lastUsedAt/lastUsedInBatchIds` — but only in `getItem`/`getAllItems`; raw IDB records may still hold old shapes until rewritten.
- **Selection thresholds/weights are hardcoded**: MIN_SCORE=3, RECENCY_DAYS=30, perf boost formula `(score-5)*0.8*min(1,sample/4)` clamped to [-3,+6], DEEP_REFERENCE_CAP=5 (deep vs breadth tier split), deep loader maxResults=10, pinned frame cap 8 vs deep primary frame cap 4 (`DEEP_PRIMARY_FRAME_CAP`), auto-star 8.0/n≥3/unstar 6.5, MAX_BATCH_HISTORY=20.
- **Auto-star vs manual star**: `autoStarred:true` marks provenance; auto-demotion only touches auto-starred items ("manual stars are sacred"). Starred items get +2 in selection regardless of origin.
- **Injection never sends raw media except frames of the top pick**: the light block is text-only; the deep block includes full reference script text (`attachedScriptText || textContent`) for the top 5 — a deliberate token-cost decision. Frames go only to the PRIMARY pick (deep) or the pin.
- **Analyzer JSON parsing is defensive but throws**: fences stripped, first-`{`/last-`}` extraction; missing `tags` or invalid JSON throws → callers must `markFailed` (InspirationBank.tsx does).
- **`reanalyzeItem` preserves user/performance state manually** — if you add new persisted fields to `InspirationItem`, you must add them to BOTH `preserved` spreads in `inspirationAnalyzer.ts` or re-analysis will wipe them.
- **`userCategories` is UI-only** (folder tabs in the bank); the selector/injection/analyzer all ignore it — documented in the type's comment.
- **`recordInspirationUsage` credits ALL injected items equally**: every deep pick (up to 10) gets the brief's full review score, not just the ones actually mirrored; runs sequentially to avoid IDB transaction races on the same item.
- **Auto-naming rule**: `titleIsFilenameLike()` (empty, equals filename, or ends in a media/doc extension) gates both `persistAnalysis` title adoption and the backfill pass — user-set titles are never overwritten.
- **frameExtractor constraints**: clamps to 2–16 frames, requires Canvas 2D, `URL.createObjectURL` + revoke; duration falls back to 1s if the browser reports non-finite duration. Frames are stored WITH the `data:image/jpeg;base64,` prefix; `stripDataUrlPrefix` (analyzer) and regex matches (pipelineEngine vision builders) strip it at call time.

# 6. The Factory (Autopilot Pipeline)

**Scope:** `src/autopilot/`

## Overview

`src/autopilot/` is the multi-agent brief-generation pipeline ("Autopilot") of the Viasox Marketing Intelligence SPA. Its job: take a batch of creative tasks — parsed from an Asana board screenshot via Claude Vision, or built manually — and turn each one into a finished, editor-ready video-ad brief (markdown with STRATEGY / SCRIPT (HOOKS) / SCRIPT (BODY) / EDITING INSTRUCTIONS tables), plus an 11-criterion quality review per brief. Output is markdown rendered in the UI and exportable as .doc/.csv from the Memory Vault; the subsystem does NOT write back to Asana — "asanaMapper" only maps Asana screenshot data INTO typed pipeline params.

The pipeline is a two-phase, human-in-the-loop orchestration (`pipelineEngine.ts`, ~1,900 lines). Phase 0: an optional Memory Curator (runs if ≥1 prior batch exists) synthesizes all creative history into a 500-800 word "Creative Intelligence Briefing" injected into every downstream agent; then a Strategy Session agent analyzes the batch and asks the creative director multiple-choice questions, whose answers are synthesized into a weekly strategy brief. Phase 1 (`runConceptPhase`), per task: (1) Creative Strategist writes a per-brief creative thesis; (2) Concept Generator produces 5 concepts (angles prompt + thesis as "#1 directive" + angle language bank + duration-specific word budgets); (3) Differentiation Critic grades each concept KEEP/REJECT — an overall REGENERATE verdict triggers exactly one regeneration with the critic's guidance; (4) Concept Evaluator rates and summarizes the concepts into `ConceptOption[]` for interactive user review. The user approves one concept per task in the UI. Phase 2 (`runScriptPhase`): the Script Writer executes the chosen concept into the unified Ecom brief template (forceEcomTemplate=true; AGC/Static keep their own shapes), validated by `validateScriptResult` (≥300 chars + must contain a markdown table). Then a Batch Reviewer scores each brief one-at-a-time (a deliberate fix — a single all-briefs call overflowed token budgets on 8+ briefs) against 11 criteria with a "rising-bar" calibration block. Finally the batch is committed to memory and the learning loops run.

Three model tiers (from `src/config/models.ts`): IDEATION_MODEL (strategist, concept generator, differentiation critic — targeted at claude-fable-5 but currently pinned to claude-opus-4-8 because the API key lacks Fable access), CREATIVE_MODEL (script writer, evaluator, selector, strategy session, curator, batch reviewer — claude-opus-4-8; aliased as `OPUS` inside files), and UTILITY_MODEL (screenshot parser, angle-directive proposer — claude-sonnet-4-6). Every task also draws inspiration context: either a user-pinned Inspiration Bank item (up to 8 video frames sent via vision calls, with hard framework/hook-style locks) or an automatic "deep" match (top-10 scored items, top pick contributes up to 4 frames).

The self-improvement layer is the subsystem's signature feature. After every batch, `memoryExtractor.saveCompletedBatchToMemory` distills each brief into a compact `BriefMemoryRecord` (framework, hook styles classified by regex, persona/emotion extracted by keyword matching, review score, full brief markdown) stored in localStorage. Four learning loops then run: (1) each used inspiration item's derivedScore is credited/blamed with the brief's review score (and high performers auto-starred); (2) `anglePatternMiner` recomputes an (angle × product × framework × hookStyles) performance table injected into future evaluators as "PROVEN WINNERS / KNOWN UNDERPERFORMERS"; (3) `scoreCalibration` recomputes rolling p25/median/p75 so the reviewer's bar rises each batch; (4) `angleDirectiveProposer` (Sonnet) mines clusters of ≥3 user redo events per angle+product for recurring preferences and drafts permanent directives the user can approve into `customOptionsRegistry`.

## Data flow

Entry point is the UI module `src/components/modules/AutopilotBriefs.tsx`, which drives the state machine defined in `src/engine/autopilotTypes.ts` (`BatchPhase`: idle → parsing → confirming → strategy-session → strategy-synthesizing → generating-concepts → concept-review → running → reviewing → complete; per-task `PipelineStep`: pending → strategist-thinking → generating-concepts → critiquing-concepts → [regenerating-concepts] → selecting-concept → awaiting-concept-approval → generating-script → complete | error). Flow: screenshot → `parseAsanaScreenshot` (Vision) → user confirms in PlannerView → `mapAsanaTask` per row → `runMemoryCurator` (if history exists) → `runStrategySession` (analysis + questions) → user answers → `synthesizeStrategy` (weekly strategy brief) → `runConceptPhase` → user picks one concept per task in ConceptReview → `runScriptPhase` → BatchResultsView. All progress flows through `onProgress(state)` callbacks that clone and re-emit `AutopilotState`; cancellation via `AbortSignal` (throws 'Pipeline cancelled').

Every Claude call is composed as: base prompt (from `src/prompts/*`) + resource context + direction block (weekly strategy brief + creative director instructions) + thesis/angle directives + optional brain addendum (`src/brain/contextAssembler.buildBrainAddendum` — a no-op empty string when the per-module flag is off; all pipeline calls share one brain session via `ensureCurrentBrainSession`, cleared at batch start and end). Pinned/deep inspiration frames are sent as base64 image blocks via `sendVisionMessage`; text-only calls via `sendMessage` (both in `src/api/claude.ts`, routed through the Cloudflare Worker CORS proxy). `sendMessageWithRetry` retries up to 3x on timeout/overload/rate-limit with 15-20s backoff; both phases also auto-retry failed tasks once after a 30s cooldown (concept-phase retry skips the critic-regeneration branch).

Outputs and state: per task — `strategistThesis`, `conceptsRaw`, `critiqueRaw`, `conceptOptions[]`, `selectedConceptText`, `recommendedFramework`, `scriptResult` (the deliverable brief markdown), `inspirationIdsUsed`. Batch-level — `reviewResult` (concatenated per-brief review sections with normalized "Brief N" headings). Persistence: `saveCompletedBatchToMemory` → localStorage (`viasox_creative_memory`); inspiration usage scores → IndexedDB via `src/inspiration/inspirationStore`; approved angle directives → localStorage via `src/utils/customOptionsRegistry` (read back into concept/script prompts on the next batch as "Custom Directives"). `redoSingleTask` re-runs the entire strategist→concept→critique→select→script chain for one task with the user's feedback as top directive, logs a `RedoEvent`, and uses `conceptSelectorPrompt` (auto-select) instead of the interactive evaluator.

## Gotchas

- The constant named `OPUS` in pipelineEngine.ts is `CREATIVE_MODEL`, but in creativeStrategist.ts and differentiationCritic.ts the same name `OPUS` is bound to `IDEATION_MODEL` — do not assume the alias means the same model across files. `IDEATION_MODEL` is currently 'claude-opus-4-8' with a TODO to flip to 'claude-fable-5' once API access is granted (models.ts:41).
- All memory is client-side localStorage under key `viasox_creative_memory` (schema version 3 in memoryStore.ts, with silent migration/backfill; corrupt JSON silently resets to an empty store). Inspiration bank data is separately in IndexedDB. Clearing browser storage destroys the entire learning system — export via `exportMemory()` first.
- Rate-limit discipline is hardcoded: `INTER_CALL_DELAY = 8000` ms between Claude calls, 30s cooldown before auto-retries (the log line says "15s" but the code sleeps 30_000), max 3 attempts with 15-20s backoff in the retry wrappers. Removing these will hit API rate limits.
- The batch reviewer intentionally runs ONE brief per call with max_tokens=6000 — a single all-briefs call previously truncated mid-JSON on 8+ briefs, making every brief show "Not scored". Do not "optimize" it back into a single call (comment at pipelineEngine.ts:1382-1393).
- Pinned inspiration is authoritarian by design: a pin's tagged framework hard-overrides the evaluator/selector's `recommendedFramework` on every concept option AND the script phase's `lockedFramework`; pinned tasks are excluded from batch framework-diversity tracking. Frame caps: 8 for pins, 4 for deep matches (`DEEP_PRIMARY_FRAME_CAP`).
- `matchFramework` fuzzy-matches free text against the 20-entry `VALID_FRAMEWORKS` list and silently falls back to 'PAS (Problem-Agitate-Solution)' — an unknown framework string never errors, it just becomes PAS.
- `validateScriptResult` treats <300 chars or no '|' character as failure; the model occasionally returns preamble-only output under load and this is the only guard against marking such tasks 'complete'.
- Scoring math: weighted composite gives hookQuality/confusionFactor/scriptLineStrength 1.5x weight (reviewParser.computeWeightedScore); criterion result thresholds are score ≤4 FAIL, ≤6 FLAG, else PASS; `inspirationAdherence` is null (excluded) when no inspiration was provided. User overrides via `updateBriefScore` overwrite `finalScore` AND `reviewScore` and feed all learning loops.
- Hook-style classification, persona extraction, and emotional-entry extraction in memoryExtractor.ts are pure keyword/regex heuristics over brief text — not model calls. Changing brief template headings (e.g. "SCRIPT (HOOKS)") breaks `extractHookSummaries` and briefMeta's table parsers, which match those literal section names.
- `AD_TYPE_OPTIONS` in asanaMapper.ts must stay mirrored with the `AdType` union in `src/engine/types.ts` (explicit comment). Ad type resolution defaults to 'Ecom Style' unless explicitly set (user requirement); the script phase always passes `forceEcomTemplate=true` — only AGC and Static keep their own output shapes (flag auto-ignored for those inside buildScriptPrompt).
- screenshotParser instructs the vision model to preserve task names verbatim ("SOX-374_ES" style) because download filenames derive from them; product abbreviations ACS/COMP/ES are translated at extraction time AND re-checked in asanaMapper.mapProduct (whole-token match before substring fallback — 'ACS' would otherwise mis-route).
- Hardcoded business values in asanaMapper: offer 'B2G3', hookVariations 3, bookReference 'All Four Books', long persona strings ("Beth the Quiet Fighter", "Linda the Practical Optimist") keyed off angle keywords. (funnelStage is no longer hardcoded 'TOF' — since July 2026 it derives from the awareness level, which is user-overridable per task in PlannerView; the awareness-conditional offer-voicing rules live in awarenessGuide + the reviewer's batch-composition block.)
- Brain addenda are strictly additive: when a module's flag is off, `buildBrainAddendum` returns an empty string, so removing those calls changes nothing except when the flag is on. One brain session spans the whole batch (created in runStrategySession, ended at runScriptPhase completion; a stale session from an interrupted run is cleared at the next runStrategySession).
- Redo events cap at 200 (MAX_REDO_EVENTS); directive proposals dedupe same angle+product+pattern within 7 days; proposer needs ≥3 redo events per cluster and analyzes at most the last 12.
- Angle pattern prompt formatting uses MIN_SAMPLE_FOR_PROMPT = 1 (a single data point is shown as signal) but PROVEN/UNDERPERFORMER verdicts require sampleSize ≥2; calibration needs ≥5 scored briefs before the rising-bar block appears.

# 7. Infrastructure: API Clients, Workers, Scripts, Deploy

**Scope:** `src/api/`, `src/config/`, `src/utils/`, `proxy/`, `meta-proxy/`, `scripts/`, `.github/`

## Overview

This subsystem is the "plumbing" layer of the Viasox Marketing Intelligence SPA: all external I/O (Claude API, Meta Graph API), the two Cloudflare Workers that broker that I/O, the model-tier configuration, the client-side auth gate, browser-side parsing/export utilities, the offline data-preprocessing scripts that bake customer data into the bundle, and the GitHub Pages build/deploy pipeline.

Claude API access works on a proxy-first, direct-fallback pattern. `src/api/claude.ts` exposes three request functions — `sendMessage` (single-turn), `sendVisionMessage` (mixed text+base64-image content blocks for screenshot parsing), and `sendChatMessage` (multi-turn history for the persona chat) — all POSTing the raw Anthropic Messages API body to `https://viasox-claude-proxy.workers.dev` with the user's key in `x-api-key`. If the proxy fetch throws, it retries against `https://api.anthropic.com/v1/messages` directly with the `anthropic-dangerous-direct-browser-access: true` header (works because Anthropic allows browser CORS with that header). The proxy itself (`proxy/worker.js`) is a ~60-line stateless CORS pass-through: it holds no secrets, forwards the caller's `x-api-key`, adds `anthropic-version: 2023-06-01`, and stamps `Access-Control-Allow-Origin: *`. Model selection is centralized in `src/config/models.ts` as three exported constants: `IDEATION_MODEL` (currently `claude-opus-4-8`, with a documented TODO to flip to `claude-fable-5` once the API key gets Fable access), `CREATIVE_MODEL` (`claude-opus-4-8`) and `UTILITY_MODEL` (`claude-sonnet-4-6`). Rollback/upgrade of any tier is a one-line change in that file.

Meta integration is deliberately asymmetric to the Claude proxy: the second Worker (`meta-proxy/worker.js`, deployed at `https://viasox-meta-proxy.kiavashmoh.workers.dev`) is stateful and secret-holding. It runs the full Facebook OAuth dance (CSRF state nonces in KV with 10-min TTL, short-lived → 60-day long-lived token exchange), stores the user token plus a per-Page token cache in Cloudflare KV, and exposes a strictly read-only `/meta/graph` proxy — non-GET Graph methods are rejected with 405 at the infrastructure level, so even write-capable OAuth scopes (`business_management` is requested purely for BM page enumeration) cannot mutate anything. The browser never sees any Meta token; `src/api/metaProxy.ts` is the typed client that only ever receives connection status, page id/name lists, and proxied Graph JSON. Page access tokens (required for `/POST_ID/comments`) are discovered via a four-strategy cascade (`/me/accounts`, business-graph traversal, `/me/assigned_pages`, per-page fallback) and cached server-side in KV.

The app is gated by a client-side team password (`src/utils/hash.ts` compares the SHA-256 of the input to a hardcoded hex digest via WebCrypto — plaintext is not in the repo) followed by an Anthropic API key screen that persists the key only in `sessionStorage` under `claude_api_key`. Two offline Node scripts in `scripts/` pre-crunch massive Shopify CSV exports (3.5M order rows streamed, 128K reviews) into `src/data/reviewAnalysis.json` and `src/data/salesEnrichment.json`, which are statically imported at build time — so the deployed SPA ships with the full segment/sales intelligence baked in and never fetches customer data at runtime. Deployment is a plain GitHub Pages workflow: push to main → `npm ci` → `tsc -b && vite build` → upload `dist/` → `actions/deploy-pages`.

## Key files

- `src/api/claude.ts` — All Claude API calls: `sendMessage`, `sendVisionMessage` (vision/screenshots), `sendChatMessage` (multi-turn). ## Data flow

Claude path: UI module builds a system+user prompt → calls `sendMessage`/`sendVisionMessage`/`sendChatMessage` (directly or via the `useClaudeApi` hook in `src/hooks/useClaudeApi.ts`, which manages loading/error state and cancels in-flight requests via AbortController) → POST to the Claude proxy Worker with the sessionStorage API key in `x-api-key` → Worker forwards to `api.anthropic.com/v1/messages` → JSON back → first `content` block of `type:'text'` is extracted and returned as a markdown string. That markdown then flows into the parsing utils: `conceptParser.parseConceptBlocks` splits concept lists; `downloadUtils.parseKvTable`/`parseScriptTable` + `normalizeBodyRow` turn brief markdown into .doc/.csv exports via Blob + programmatic `<a>.click()` downloads. Error semantics in `sendMessage`: proxy fetch throws → direct API fallback; HTTP 429/529 → up to 8 retries with exponential backoff (15s doubling, capped 180s, +0–5s jitter) — retries go through the PROXY only; 401 → "Invalid API key"; timeout is model-based and ignores max_tokens (Opus: fixed 30 min, everything else: fixed 15 min); `stop_reason === 'max_tokens'` appends a visible truncation note to the returned text.

Meta path: user clicks Connect → `startMetaOauth` does a full-page redirect to the Worker's `/meta/oauth/start` (return_to validated against the origin allowlist, CSRF nonce stored in KV for 600s) → Meta authorize page → Worker `/meta/oauth/callback` exchanges code → short-lived → 60-day token, fetches `/me` identity, stores `{access_token, expires_at, user_*}` in KV under `meta:token`, redirects back with `?meta=connected`. Thereafter the browser calls `getMetaStatus` (token never returned; only identity + `days_remaining`), `refreshPageTokens` (populates KV `meta:page_tokens` via the four-strategy discovery, returns counts + `failed_pages` needing BM Task assignment), and `metaGraph({path, params, page_id?})` → Worker `/meta/graph` injects the right token (page token if `page_id` given, else user token; 412 if the page token isn't cached) and performs a forced-GET Graph call. Consumers: `src/comments/commentPuller.ts` and `src/components/comments/*` (MetaConnectionPanel, CommentPullPanel).

App/auth state: `App.tsx` runs a view state machine (auth → apikey → upload → processing → dashboard → module). PasswordGate → `verifyPassword` (SHA-256 compare, purely client-side, not persisted — refresh requires re-entry) → ApiKeyInput validates the `sk-ant-` prefix and writes `sessionStorage['claude_api_key']`; `App.tsx` seeds its `apiKey` state from that same key so it survives reloads within a tab session but not new tabs/windows.

Offline/build data flow: `node scripts/preprocess-reviews.mjs` and `node scripts/preprocess-sales-data.mjs` read CSVs from the sibling directory `../VSX Intelligence CSV Data Files/` (auto-detected by filename prefix: `Orders*`, `Customer*`, `Viasox Reviews*`), stream the huge files via PapaParse step callbacks, and write `src/data/reviewAnalysis.json` and `src/data/salesEnrichment.json`. Those JSONs are statically imported by `src/data/reviewAnalysisLoader.ts` / `salesEnrichmentLoader.ts`, so they are bundled into the deployed JS — regenerating the data requires rerunning the scripts and rebuilding.

Deploy flow: push to `main` (or manual dispatch) → `.github/workflows/deploy.yml` → Node 20, `npm ci`, `npm run build` (typecheck via `tsc -b` then `vite build` with base `/Viasox-marketing-intelligence/`) → `dist/` uploaded as the Pages artifact → deployed. The two Workers are deployed separately and manually via `npx wrangler deploy` from their respective folders (the Meta Worker additionally needs `wrangler kv namespace create META_KV` and the three secrets set before first deploy).

## Gotchas

- Hardcoded proxy URLs: `PROXY_URL = 'https://viasox-claude-proxy.workers.dev'` in `src/api/claude.ts` has NO account subdomain (compare `viasox-meta-proxy.kiavashmoh.workers.dev` in metaProxy.ts). If that hostname doesn't resolve, every call silently falls back to the direct Anthropic API path — the app still works, but the 429/529 retry loop (which only retries via the proxy) effectively becomes single-shot. Changing Worker names/accounts requires editing both client files by hand.
- `src/config/models.ts` is the ONLY place model IDs live. `IDEATION_MODEL` is temporarily `claude-opus-4-8` because the API key lacks `claude-fable-5` access (API returns 404 for it); flip the one line back when access is granted. A bad ID surfaces as an immediate "400 invalid model".
- `computeTimeout` ignores its `maxTokens` argument despite the doc comment: any model whose ID contains "opus" gets a fixed 30-minute timeout, everything else 15 minutes.
- The Claude proxy Worker allows `Access-Control-Allow-Origin: *` — any site can relay through it (it holds no secrets; the caller supplies the key). The Meta Worker, by contrast, enforces the `APP_ORIGIN_ALLOWLIST` secret (comma-separated origins, e.g. GitHub Pages origin + `http://localhost:5173`) on all browser-facing endpoints; adding a new dev/prod origin means updating that Worker secret, not code.
- Meta read-only guarantee is enforced by the Worker rejecting non-GET Graph methods (405), NOT by scopes — `business_management` nominally grants write ability but is physically write-blocked. Don't "fix" `metaGraph` to support POST.
- Meta tokens: single-tenant token vault — one KV key `meta:token` for the whole app (whoever OAuths last wins), 60-day expiry with no auto-refresh (`days_remaining` in status is the signal). Page tokens must be refreshed via `/meta/page-tokens/refresh` before any `page_id` Graph call, else 412. The Sunday cron in wrangler.toml is wired but the handler is a placeholder (writes `meta:cron:last_run` only).
- Password auth is cosmetic client-side gating: SHA-256 hash is in the bundle, all data too. Plaintext is not in the repo (ask the user if needed). Password is re-required per page load; only the API key persists (sessionStorage, per-tab).
- The segment/pain/benefit/transformation regexes are DUPLICATED VERBATIM in three places: `src/engine/patterns.ts` (runtime), `scripts/preprocess-reviews.mjs`, and `scripts/preprocess-sales-data.mjs`. Changing a pattern requires updating all three and regenerating both JSONs, or runtime and baked data disagree.
- The preprocessing scripts hard-depend on the sibling folder `../VSX Intelligence CSV Data Files/` (outside the repo, matched by filename prefixes) and on Shopify column names ('Customer email', 'Net sales', 'Product type', etc.); `PRODUCT_TYPE_MAP` in preprocess-sales-data.mjs hardcodes the Shopify product-type → 3-product-line mapping.
- Case-sensitivity trap: vite base is `/Viasox-marketing-intelligence/` (capital V) but index.html's favicon href hardcodes lowercase `/viasox-marketing-intelligence/vite.svg` — the favicon 404s if the Pages path is capital-V; keep the base in sync with the actual GitHub repo name.
- `downloadUtils.normalizeBodyRow` encodes a hard invariant: the spoken line is always the RIGHTMOST table cell, visual second-to-last. Prompt-template changes that move columns will be silently "corrected" by this right-anchoring — check the `console.warn` drift messages when debugging empty script lines.
- `downloadBriefForAdType` routes `'AI Podcast'` in its switch, but that value is not in `BASE_AD_TYPES` (customOptionsRegistry) — ad-type strings are matched by exact literal equality (e.g. `'AGC (Actor Generated Content)'`), so renaming an option breaks the export router.
- Local builds on this machine need Node from `~/local/node`: prefix `PATH="/Users/kiavashmohammadi/local/node/bin:$PATH"` before any npm command. `npm run build` runs `tsc -b` first, so type errors block the build (and thus CI deploys).
- README.md is untouched Vite boilerplate — do not treat it as project documentation.
